What is SendHub.ai?

SendHub.ai is a multi-tenant SaaS platform that gives businesses AI-powered WhatsApp messaging with a unified inbox, intelligent auto-replies, team collaboration, and e-commerce integrations. It supports both WhatsApp Business API (cloud-hosted) and personal WhatsApp accounts via a Matrix bridge.

The platform includes an embeddable web chat widget, a knowledge base with semantic search, Shopify product/order integration, scheduled messaging, and an MCP server for AI agent access.

Core Principle: "Client Never Delivers"

The dashboard never calls WhatsApp or Matrix APIs directly. It only inserts messages into the database with delivery_status: 'queued'. All delivery is handled server-side by the ai-worker, enabling centralized retry logic and clean separation of concerns.

Deep Dive

Why Supabase enables rapid, error-free development

Supabase isn't just a database — it's the entire backend layer. By making PostgreSQL the single source of truth and leveraging its built-in services, we eliminated entire categories of bugs and boilerplate.

1 Row-Level Security — Security at the DB, not the app

Every table has RLS policies enforced at the PostgreSQL level. Users can never see data from other organizations, regardless of what the application code does. This eliminates an entire class of authorization bugs.

Multi-tenant hierarchy

Organization (billing) → Business (department) → Channel (WA number) → Team Member

Hard boundary on org_id, soft boundary on business_id. Helper functions: get_my_team_org_ids(), get_my_business_ids()

2 PostgREST — No hand-written CRUD

Supabase auto-generates a REST API from the database schema. Every table gets typed queries, filters, and joins out of the box. The Hono Worker only handles business logic (webhooks, AI orchestration, delivery) — not basic CRUD.

Result: Estimated 10-12 days saved vs. hand-writing a custom API. Zero bugs from serialization mismatches.

3 Realtime — Live dashboard, zero custom WebSockets

Supabase Realtime broadcasts database changes over WebSocket. When a new message is inserted (by webhook, AI worker, or agent), the dashboard receives it instantly. No custom WebSocket server, no message broker, no polling.

Also powers the "Client Never Delivers" pattern: the ai-worker subscribes to message inserts and handles delivery, triggered by the same Realtime channel.

4 Auth — JWT + RLS integration

Supabase Auth issues JWTs that are automatically validated by RLS policies. Google OAuth is a config toggle. Signup auto-creates org, default business, and team member via a database trigger. API keys use hashed storage with scoped access.

5 pgvector — Semantic search in the same DB

Knowledge base documents are chunked, embedded via Cloudflare Workers AI (free, 768d), and stored in pgvector. Semantic search queries run in ~1ms on IVFFlat indexes. No external vector DB dependency — everything stays in the SSOT.

SQL RPCs: search_knowledge_vector(), search_knowledge_text(), search_conversation_messages()

6 Storage — RLS-enforced media

WhatsApp media (images, video, documents) are downloaded from Meta CDN and stored in Supabase Storage with RLS policies. Private buckets ensure org-scoped access. No separate S3 config needed.

Cascading AI Prompts

A single business can have multiple AI personalities per channel. The prompt resolution cascades: channel-specific override beats business-level prompt, which beats the org default.

Example: Acme Retail has two channels on the same number:

  • • Sales channel: Aggressive, upsell-focused AI persona
  • • Support channel: Empathetic, help-first AI persona

Same infrastructure, different personalities. No duplication.

Development Process

AI-augmented development with Claude Code

SendHub.ai was built with extensive use of Claude Code for AI-augmented development. This isn't about generating boilerplate — it's about having an AI pair programmer that understands the full architecture and can make informed decisions across the stack.

1 Architecture-aware code generation

Claude Code has full context of the codebase via CLAUDE.md files documenting architecture decisions, conventions, and the multi-tenant model. New features are generated with correct RLS patterns, proper error handling, and consistent API design from the start.

2 Database migrations & RLS policies

SQL migrations with RLS policies are complex and error-prone. Claude Code generates migrations that follow established patterns — denormalized org_id/business_id columns, proper policy functions, and cascade-aware foreign keys — catching security gaps before they reach production.

3 Cross-service consistency

With 6+ deployed services (Worker, Dashboard, Admin, Shopify App, Matrix Socket, MCP Server), keeping interfaces consistent is critical. Claude Code ensures type alignment across services, consistent error handling patterns, and shared conventions like the "Client Never Delivers" rule.

4 Rapid feature development

Features like the knowledge base semantic search, scheduled messaging, Shopify webhook sync, and email channel integration were implemented end-to-end (migration + API + UI) in single sessions. The AI understands the patterns and produces production-ready code that follows the project's conventions.

5 Documentation as development

Comprehensive CLAUDE.md files serve dual purpose: they guide AI-augmented development and document the system for human developers. Architecture decisions, integration flows, deployment procedures, and debugging guides are maintained as living documentation.

Decisions

Key technical decisions

Database-first architecture

Supabase PostgreSQL as SSOT. PostgREST auto-generates the API. RLS enforced at DB level, not app middleware. Realtime subscriptions for live updates. Auth + JWT built-in. Trade-off: Less flexibility than hand-written API, but massive productivity gain.

Stateless Workers for the hot path

Webhook processing and AI replies run on Cloudflare Workers — zero cold start, global edge, low latency. Meta webhook timeout is 30s, so fast processing is critical. Persistent services (Matrix socket, Shopify app) only for long-lived connections.

pgvector over external vector DBs

Semantic search with pgvector IVFFlat index (~1ms queries on 10K chunks). Stored in the SSOT. Supports cascading scopes (org/business/channel). Alternatives considered: Pinecone, Weaviate — added external dependency and costs scale with embedding count.

Multi-tenant hierarchy from day 1

Org → Business → Channel → Team Member. Enterprises: 1 org, many businesses. SMBs: 1 org, 1 business (hidden layer). AI prompts cascade through the hierarchy. Scales from solo pizza shop to enterprise with 100 departments.

AI provider abstraction

Pluggable interface: resolveProviderConfig(env, orgSettings) → callProvider(config, input). Customers choose provider (cost, preference). Fallback if one provider is down. Cost arbitrage (Gemini cheapest, Claude highest quality).
